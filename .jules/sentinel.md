## 2024-05-28 - Insecure Storage of Plaintext Passwords
**Vulnerability:** The Password feature was saving user-entered credentials in plaintext directly into `AsyncStorage`. This is extremely insecure because `AsyncStorage` writes to an unencrypted XML/JSON file on the device (or SharedPreferences on Android), which can be easily dumped by root users or malicious apps, potentially compromising highly sensitive HR portal and airline service passwords.
**Learning:** React Native developers often conflate "local storage" with "secure storage." While `AsyncStorage` is convenient for preferences and non-sensitive data, any user secrets must be placed into the platform's keychain/keystore. In this codebase, although a PIN feature was present and the PIN itself was stored securely via `SecureStore`, the actual payload (the passwords) was left exposed in plain text. A hybrid storage pattern where the non-sensitive metadata (username, notes, ID) resides in `AsyncStorage` while the secrets (password) are delegated to `SecureStore` (e.g. key: `aerostaff_pwd_${id}`) solves the 2048-byte `SecureStore` size limit problem while enforcing strict security.
**Prevention:** Whenever designing a feature that handles credentials, API keys, or sensitive PII, default to using `expo-secure-store` or equivalent keychains. If the data is an array or object that exceeds size limits, adopt the hybrid pattern immediately instead of resorting to `AsyncStorage` for the whole object. Ensure proper migration logic is added for users with legacy insecure data.
